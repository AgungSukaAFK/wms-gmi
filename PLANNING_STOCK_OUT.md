# Planning: Fitur Stock Out (SPB Flow)

> **Referensi**: WMS lama (branch `frontend` + `backend` di `AgungSukaAFK/wms-old-sample`)
> **Target Layout**: Ikuti pola halaman baru (Content wrapper, server actions, Supabase, shadcn/ui)

---

## Overview

Kategori **Stock Out** terdiri dari 6 fitur halaman terpisah, mengikuti alur dokumen WMS lama:

```
SPB ──attach──▶ SPB-PO ──attach──▶ SPB-DO ──attach──▶ Invoice
 │
 └──return──▶ Return SPB (mengembalikan stok)

Report SPB = Laporan gabungan flat (SPB + PO + DO + Invoice)
```

### Daftar Halaman & Route

| No  | Menu Sidebar   | Route          | Deskripsi                                    |
| --- | -------------- | -------------- | -------------------------------------------- |
| 1   | Report SPB     | `/spb/report`  | Tabel flat report gabungan SPB→PO→DO→Invoice |
| 2   | SPB            | `/spb`         | CRUD Surat Pengeluaran Barang (kurangi stok) |
| 3   | Purchase Order | `/spb/po`      | Attach PO ke SPB (bukan PO procurement)      |
| 4   | Delivery Order | `/spb/do`      | Attach DO ke SPB-PO                          |
| 5   | Invoice        | `/spb/invoice` | Attach Invoice ke SPB-DO                     |
| 6   | Return SPB     | `/return-spb`  | Return barang dari SPB (tambah stok kembali) |

### 1.3 View untuk Report SPB

```sql
CREATE OR REPLACE VIEW public.v_spb_report AS
SELECT
    s.id AS spb_id,
    s.spb_no,
    s.spb_tanggal,
    s.spb_no_wo,
    s.spb_section,
    s.spb_pic_gmi,
    s.spb_pic_ppa,
    s.spb_kode_unit,
    s.spb_tipe_unit,
    s.spb_brand,
    s.spb_hm,
    s.spb_problem_remark,
    s.spb_status,
    s.spb_gudang,
    s.created_at AS spb_created_at,

    sd.id AS spb_dtl_id,
    sd.dtl_spb_part_number,
    sd.dtl_spb_part_name,
    sd.dtl_spb_qty,
    sd.dtl_spb_part_satuan,

    po.po_no,
    po.so_no,
    po.created_at AS po_created_at,

    do_.do_no,
    do_.created_at AS do_created_at,

    inv.invoice_no,
    inv.invoice_date,
    inv.invoice_email_date,
    inv.created_at AS invoice_created_at

FROM public.spb s
JOIN public.spb_details sd ON sd.spb_id = s.id
LEFT JOIN public.spb_po_details pod ON pod.spb_dtl_id = sd.id
LEFT JOIN public.spb_po po ON po.id = pod.spb_po_id
LEFT JOIN public.spb_do_details dod ON dod.spb_po_dtl_id = pod.id
LEFT JOIN public.spb_do do_ ON do_.id = dod.spb_do_id
LEFT JOIN public.spb_invoice_details invd ON invd.spb_do_dtl_id = dod.id
LEFT JOIN public.spb_invoice inv ON inv.id = invd.spb_invoice_id
WHERE s.spb_is_deleted = FALSE;
```

## FASE 3: Server Actions

File: `services/spb-actions.ts` (file baru)

### 3.1 SPB Actions

| Fungsi                                                   | Deskripsi                           | Input              | Output                   |
| -------------------------------------------------------- | ----------------------------------- | ------------------ | ------------------------ |
| `getSpbList(search?, cabangId?, status?, page?, limit?)` | List SPB + pagination               | filter params      | `{ data: Spb[], count }` |
| `getSpbByNo(spbNo: string)`                              | Detail SPB by kode                  | spb_no             | `Spb` with details       |
| `getSpbById(id: number)`                                 | Detail SPB by id                    | id                 | `Spb` with details+po    |
| `createSpb(payload)`                                     | Create SPB + items, decrement stock | header + details[] | `Spb`                    |
| `updateSpb(id, payload)`                                 | Update SPB header (e.g. no_wo)      | partial fields     | `Spb`                    |
| `deleteSpb(id)`                                          | Soft delete SPB, restore stock      | id                 | `void`                   |
| `generateSpbKode(cabangId)`                              | Generate kode SPB auto-increment    | cabang_id          | `string` kode            |

**Logika penting `createSpb`:**

- Validasi stok mencukupi di cabang
- Decrement stok (`stock.qty -= dtl_spb_qty`) per item
- Insert SPB header + details
- Jika stok tidak cukup, throw error

**Logika penting `deleteSpb`:**

- Soft delete (set `spb_is_deleted = true`)
- Restore stok: increment stock.qty per detail item

## Catatan Teknis Penting

### Pola yang HARUS diikuti (dari WMS baru):

1. **Server Actions** — Semua backend logic pakai `"use server"` di `services/spb-actions.ts`
2. **Supabase client** — Gunakan `createClient()` dari `@/lib/supabase/server` di server actions
3. **Content wrapper** — Semua halaman pakai `<Content>` component
4. **Pagination** — Pakai pattern `DataTablePagination` atau custom pagination yang sudah ada
5. **Combobox** — Pakai component `Combobox` dari `@/components/combobox.tsx` untuk search items
6. **Date picker** — Pakai `DatePicker` dari `@/components/date-picker.tsx`
7. **Detail Sheet** — Pakai `Sheet` dari shadcn/ui untuk side panel detail
8. **Debounced search** — 500ms delay pakai `setTimeout`
9. **Error handling** — `try/catch` + `toast.error()` dari sonner
10. **Revalidation** — Setelah mutasi, panggil `revalidatePath("/(With Sidebar)/spb")`

### Pola dari WMS lama yang HARUS dipertahankan:

1. **Stock decrement saat create SPB** — Kurangi stok di cabang/gudang saat SPB dibuat
2. **Stock increment saat return** — Tambah stok kembali saat return SPB
3. **Status chain**: `DONE QUOT` → `PO_ATTACH` → `DO_ATTACH` → `DONE_QUOTE`
4. **Kode auto-generate** — Format: `GMI{KODE_LOKASI}/{YYYY}/{MM}/{SEQUENCE}` untuk SPB, `RTN/{YYYY}/{MM}/{SEQUENCE}` untuk Return
5. **Soft delete** — SPB pakai `spb_is_deleted` flag, bukan hard delete
6. **Cascading relationship** — SPB → PO → DO → Invoice, setiap step "attach" document berikutnya
7. **Return qty validation** — Max return per item = `dtl_spb_qty - dtl_spb_qty_returned`

### Yang TIDAK perlu diimplementasi (legacy):

- Export Excel (bisa ditambahkan nanti)
- Export PDF (bisa ditambahkan nanti)
- Signature flow di SPB (WMS lama juga belum ada di SPB)
- Closing Book check (legacy feature, skip)
